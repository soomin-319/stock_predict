# 주가 예측 파이프라인(v14) 전문가 코드 리뷰

## 한 줄 총평
현 코드베이스는 **"평균회귀로 인한 예측 변동폭 붕괴"라는 시계열 회귀의 핵심 문제를 정확히 진단**했고, 이를 완화하기 위한 설계(변동성 정규화 + 사후 보정 + 크기 가중 손실)가 논리적으로 잘 연결되어 있습니다. 다만, **검증/테스트 누수 가능성, 목적함수-의사결정 불일치, 시계열 검증 프로토콜 부재**가 남아 있어 실전 성능 신뢰도를 높이려면 구조적 보강이 필요합니다.

---

## 강점 (잘한 점)

### 1) 문제 정의가 정확함: "E[r|X] ≈ 0" 압축 문제 대응
- 타겟을 `raw_return / daily_vol`로 바꾼 것은 **heteroskedasticity(조건부 분산 변화)** 대응 관점에서 타당합니다.
- 학습 후 다시 `* daily_vol`로 복원하는 흐름도 올바른 접근입니다.

### 2) 손실 함수 설계가 목적에 맞음
- `magnitude_weighted_directional_huber_loss`는
  - Huber로 이상치 안정성 확보,
  - 부호 반대 예측에 대한 direction penalty,
  - `|y_true|` 기반 가중으로 큰 변동 학습 강화
  를 결합해 의도가 분명합니다.

### 3) 누수 방지 의식이 높음
- `target_col`을 보간 대상에서 제외한 점,
- 스케일러를 train 구간으로만 fit한 점,
- time-based split을 유지한 점은 모두 좋은 패턴입니다.

### 4) 운영 관점 완성도
- 외부 시계열 캐시(parquet), 병렬 fetch, 결과 CSV/ZIP, 시드 앙상블, 디버그 파일 출력 등 재현성/운영성을 고려했습니다.

---

## 핵심 리스크 (성능 왜곡 가능 지점)

### 1) 가장 중요한 이슈: "내일 예측용 시퀀스"가 테스트 분할에 섞일 가능성
현재 `prepare_sequences_with_externals()`에서 마지막에 "오늘→내일" 시퀀스를 `y=np.nan`으로 추가합니다.
이 상태에서 곧바로 `split = int(len(X_all)*(1-test_ratio))`로 자르면 다음 문제가 생길 수 있습니다.

- `y_test_raw`에 NaN이 들어감 (메트릭에서 제외하긴 함)
- calibration/summary에서 **실제값 없는 샘플이 테스트 말단에 고정적으로 위치**
- 전략평가 시 "진짜 OOS"와 "실시간 추론용 샘플"이 섞여 해석이 흐려짐

권장:
- 학습/평가용 시퀀스와 "실시간 추론용 마지막 1샘플"을 구조적으로 분리하세요.
  - `prepare_sequences...`는 `train_eval_set`만 반환
  - `prepare_live_sequence` 별도 함수로 today→tomorrow 1개 생성

### 2) 사후 보정(calibration) 방식의 안정성
현재 `calib_scale = std(actual_val_raw) / std(pred_val_raw)`은 직관적이지만,
- 검증 샘플 수가 작으면 분산 추정이 매우 불안정,
- 극단적 구간에서 스케일이 출렁일 수 있습니다.

권장:
- 시드별 단일 스케일보다는,
  - 롤링 윈도우 기반 캘리브레이션,
  - 혹은 여러 fold에서 얻은 스케일의 중앙값 사용.

### 3) 손실 가중치가 배치 통계에 의존
`w = 1 + mag_weight * |y_true| / mean(|y_true|)`는 **현재 배치의 mean_abs**에 의존합니다.
- 배치 조성에 따라 같은 샘플의 상대 가중이 바뀜
- 학습 신호 변동성이 커질 수 있음

권장:
- train set 전체 기반 상수(예: `global_mean_abs`)를 사전 계산해 사용.

### 4) 동적 임계값 로직이 사실상 고정값
코드에선 `pred_std`를 계산하지만 임계값은 결국 `threshold = 0.005`로 고정됩니다.
즉 "동적 임계값" 의도와 실제가 다릅니다.

권장:
- 예: `threshold = max(min_th, k * pred_return_std)`
- 또는 최근 60일 예측 분포의 분위수 기반 임계값 사용.

### 5) 목표함수와 최종 의사결정의 불일치
- 학습 손실은 연속 회귀(오차 최소화)
- 최종 산출은 매수/매도/관망의 **방향성 분류 + 임계값 룰**

권장:
- 멀티태스크 구조(수익률 회귀 + 방향 분류 헤드) 또는
- 회귀값을 rank/quantile로 변환한 정책학습 레이어 추가.

## 데이터/특징 관점 점검

### 좋은 점
- `CORE_FEATURES`를 7개로 축소한 것은 과적합 억제에 유리.
- `rv_20`, `ATR14`, 모멘텀, RSI, 밴드 위치, 거래량 급등 조합은 기본적으로 균형적.

### 보완 제안
1. **정권(regime) 피처 추가**
   - 시장 변동성 상태(저/중/고), 추세 상태(상승/횡보/하락) 인코딩
2. **캘린더 피처**
   - 요일, 월말/분기말, 옵션만기일 주변 등
3. **외부 변수 lag 자동탐색 고도화**
   - 지금은 lag 기본 0. 변수별 최적 lag를 rolling-CV로 선택 권장

## 검증 프로토콜 (가장 중요)

현재 단일 holdout(최근 20%)은 빠르지만, 금융 시계열에서는 운 좋음/나쁨 편차가 큽니다.

권장 우선순위:
1. **Walk-forward validation** (expanding window)
2. 각 fold의
   - MSE/MAE (return/close)
   - hit ratio(방향 적중률)
   - 전략지표(수수료 반영 누적수익, MDD, Sharpe)
   를 분리 기록
3. "모델 성능"과 "전략 성능"을 분리 평가

## 코드 레벨 개선 체크리스트 (실행 우선순위)

### P0 (즉시)
- [ ] live 샘플(today→tomorrow)과 test 샘플 분리
- [ ] 동적 임계값 로직 실제 동작하도록 수정
- [ ] 캘리브레이션 스케일을 fold/seed 중앙값 방식으로 안정화

### P1 (단기)
- [ ] walk-forward 검증 루프 도입
- [ ] 방향성 분류 지표(hit ratio, precision/recall by sign) 추가
- [ ] 거래비용 반영 백테스트 지표 추가

### P2 (중기)
- [ ] 멀티태스크(회귀+분류) 모델 실험
- [ ] Transformer/TFT/N-BEATS 계열과 비교실험
- [ ] uncertainty 추정(MC Dropout or Quantile Regression)

## 결론
지금 코드의 v14 방향은 맞습니다. 특히 "변동폭 압축" 문제를 정면으로 다룬 것은 매우 좋습니다. 다만 실전 배치 기준에서 성능 신뢰도를 확보하려면, **(1) 평가 설계 분리(live vs test), (2) walk-forward 검증, (3) 의사결정 로직 정합성 개선**이 반드시 필요합니다. 이 세 가지만 먼저 정리해도 실제 성능 해석력이 크게 좋아집니다.
