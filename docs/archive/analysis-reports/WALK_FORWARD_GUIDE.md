# 워크포워드 검증과 OOF 예측 가이드

## 1. 한눈에 이해하기

이 프로젝트는 과거 전체 데이터를 무작위로 섞어 학습·평가하지 않는다.
실제 운영 상황처럼 **과거 데이터로 학습하고, 이후 미래 구간을 예측하는 과정**을 시간순으로 반복한다.

이를 **워크포워드(Walk-forward)** 방식이라고 한다.

```text
과거 데이터로 학습 → 바로 다음 미래 구간 예측
                  → 시간을 앞으로 이동
더 많은 과거 데이터로 학습 → 그다음 미래 구간 예측
```

워크포워드 과정에서는 두 종류의 결과가 만들어진다.

| 결과 | 의미 |
|---|---|
| 워크포워드 검증 결과 | 각 미래 구간에서 모델 성능이 얼마나 좋았는지 보여주는 성적표 |
| 워크포워드 OOF 예측 | 모델이 각 미래 구간에서 실제로 생성한 날짜·종목별 예측값 |

쉽게 비유하면:

```text
워크포워드 검증 = 여러 번의 시험 과정과 시험별 점수
워크포워드 OOF 예측 = 시험에서 모델이 작성한 답안 전체
```

---

## 2. 왜 일반적인 무작위 분할을 사용하지 않는가?

주식 데이터는 시간 순서가 중요하다.

과거와 미래 데이터를 무작위로 섞으면 모델이 미래 정보를 학습에 사용하게 될 수 있다.
이 경우 실제 운영에서는 재현할 수 없는 과도하게 좋은 성능이 나올 수 있다.

워크포워드는 항상 다음 원칙을 지킨다.

```text
학습 데이터 날짜 < 검증 데이터 날짜
```

즉, 각 검증 날짜의 예측은 해당 날짜보다 과거에 있던 정보만 사용한다.

---

## 3. 프로젝트의 전체 흐름

```text
OHLCV 및 시장 데이터
    ↓
특징 생성
    ↓
다음 거래일 목표값 생성
    ↓
시간순 워크포워드 fold 생성
    ↓
각 fold에서 새 모델 학습
    ↓
미래 검증 구간 예측
    ├─ 성능 지표 계산
    └─ OOF 예측 저장
    ↓
모든 fold의 OOF 예측 결합
    ↓
확률 보정·신호 가중치 튜닝·백테스트
    ↓
최종 모델 별도 학습
    ↓
최신 날짜의 다음 날 예측 생성
```

주요 구현 파일:

- `src/validation/walk_forward.py`: fold 생성, fold 실행, 검증 결과와 OOF 생성
- `src/config/settings.py`: 워크포워드 설정
- `src/pipeline.py`: OOF 후처리, 튜닝, 백테스트, 최신 예측
- `src/models/lgbm_heads.py`: fold별 모델 학습과 예측

## 4. 무엇을 예측하는가?

각 날짜의 특징을 사용하여 **다음 거래일 로그수익률**과 **상승 여부**를 예측한다.

```text
target_log_return = log(다음 거래일 종가 / 현재 종가)
target_up         = target_log_return > 0
```

모델은 여러 예측 결과를 동시에 생성한다.

| 출력 | 설명 |
|---|---|
| `predicted_return` | 다음 날 예상 로그수익률 |
| `up_probability` | 다음 날 상승 확률 |
| `quantile_low` | 낮은 분위수 예상 수익률 |
| `quantile_mid` | 중앙 분위수 예상 수익률 |
| `quantile_high` | 높은 분위수 예상 수익률 |

분위수 예측 범위는 예측 불확실성을 계산할 때 사용된다.

## 5. 워크포워드 fold 생성 방식

기본 학습 설정은 다음과 같다.

```python
min_train_size = 252 * 3  # 최초 학습 구간: 약 3년
test_size = 252           # 각 검증 구간: 약 1년
step_size = 126           # 다음 fold 이동 간격: 약 반년
purge_gap_days = 1        # 학습 끝부분에서 제외할 거래일 수
embargo_days = 0          # 검증 시작 전 추가 대기 거래일 수
```

현재 구현은 **확장형 학습 구간(Expanding Window)** 방식이다.

```text
Fold 1: [──── 과거 학습 ────] [간격] [──── 검증 ────]

Fold 2: [──────── 더 긴 과거 학습 ────────] [간격] [──── 검증 ────]

Fold 3: [──────────── 더 긴 과거 학습 ────────────] [간격] [──── 검증 ────]
```

학습 시작 날짜는 유지된다. 시간이 흐를수록 학습 종료 날짜가 앞으로 이동하여 학습 데이터가 누적된다.

## 6. Purge gap과 Embargo

### 6.1 Purge gap

검증 시작 직전의 일부 거래일을 학습 데이터에서 제거한다.

```text
[학습 구간] [Purge gap] [검증 구간]
```

목적은 학습 데이터의 다음 날 목표값이 검증 구간과 겹쳐 발생할 수 있는 미래 정보 누수를 줄이는 것이다.

현재 기본값은 `purge_gap_days = 1`이다.

### 6.2 Embargo

Purge gap 이후 검증 시작점을 추가로 미래 방향으로 이동시킨다.

```text
[학습 구간] [Purge gap] [Embargo] [검증 구간]
```

현재 기본값은 `embargo_days = 0`이다.

> 설정 이름에는 `days`가 있지만 현재 구현은 달력 일수가 아니라 정렬된 고유 거래일 위치를 기준으로 계산한다.

## 7. 각 fold에서 수행하는 작업

각 fold마다 새로운 `MultiHeadStockModel`을 만든다.

```text
1. 해당 fold의 과거 구간으로 모델 학습
2. 해당 fold의 미래 검증 구간 예측
3. 실제 목표값과 예측값 비교
4. fold 성능 지표 저장
5. 검증 구간의 행별 예측값을 OOF로 저장
```

### 계산되는 검증 지표

| 종류 | 지표 | 의미 |
|---|---|---|
| 회귀 | MAE | 예측 오차 절댓값 평균 |
| 회귀 | RMSE | 큰 오차에 더 큰 벌점을 주는 오차 지표 |
| 회귀 | Corr | 실제 수익률과 예측 수익률의 상관계수 |
| 분류 | Accuracy | 상승·하락 방향 정확도 |
| 분류 | ROC-AUC | 상승 종목을 상대적으로 잘 구분하는 정도 |

파이프라인 보고서의 `walk_forward` 항목은 fold별 지표를 단순 평균한 결과다.

## 8. 워크포워드 검증이란?

워크포워드 검증은 **모델을 시간순으로 반복 학습·평가하는 전체 절차**다.

주요 질문:

> 과거 데이터만 학습한 모델이 이후 미래 구간에서 어느 정도 성능을 냈는가?

현재 코드의 검증 전용 함수:

```python
walk_forward_validate(...)
```

반환 결과는 fold별 검증 기간과 성능 지표다.

```python
FoldResult(
    train_end=...,
    valid_start=...,
    valid_end=...,
    metrics=...,
)
```

## 9. 워크포워드 OOF 예측이란?

OOF는 **Out-of-Fold**의 약자다.

각 OOF 행은 해당 행을 학습에 사용하지 않은 모델이 만든 예측이다.
워크포워드에서는 해당 날짜보다 과거 데이터만 학습한 모델이 예측을 생성한다.

```text
Fold 1 검증 구간 예측
  + Fold 2 검증 구간 예측
  + Fold 3 검증 구간 예측
  = 워크포워드 OOF 예측 데이터
```

OOF 예측에는 다음 정보가 포함된다.

```text
날짜, 종목, 실제 다음 날 수익률, 실제 상승 여부,
예측 수익률, 상승 확률, 분위수 예측값
```

주요 질문:

> 과거 각 날짜에서 실제 운영했다면 모델이 어떤 예측을 만들었는가?

현재 코드의 OOF 전용 함수:

```python
walk_forward_oof_predictions(...)
```

## 10. 워크포워드 검증과 OOF 예측 차이

| 구분 | 워크포워드 검증 | 워크포워드 OOF 예측 |
|---|---|---|
| 성격 | 반복 학습·평가 절차와 성적표 | 절차에서 생성된 행 단위 예측 데이터 |
| 데이터 단위 | Fold | 날짜·종목 |
| 주요 출력 | MAE, RMSE, Corr, Accuracy, ROC-AUC | 예측 수익률, 상승 확률, 분위수 예측 |
| 주요 목적 | 모델 일반화 성능 평가 | 보정, 튜닝, 백테스트, 진단 |
| 최신 운영 예측 여부 | 아님 | 아님 |

두 결과는 서로 다른 목적을 가지지만 같은 fold 실행에서 함께 만들어진다.

```python
folds, oof = walk_forward_validate_with_oof(...)
```

이 함수는 워크포워드 학습을 한 번 실행하고 다음 두 값을 함께 반환한다.

```text
folds = 워크포워드 검증 결과
oof   = 워크포워드 OOF 예측
```

## 11. OOF 예측은 어디에 사용되는가?

### 11.1 상승 확률 보정

고유 OOF를 먼저 시간순 tune/eval로 분리한다. `IsotonicRegression` 보정기는 앞쪽 tune OOF만 사용해 학습한다.
학습된 보정기의 변환만 tune, eval, 최신 예측에 적용한다. 따라서 eval 실제 결과는 보정기 학습에 사용되지 않는다.

보고서의 `probability_calibration.tune`과 `probability_calibration.eval`에서 구간별 Brier score와 ECE를 확인할 수 있다.

### 11.2 신호 점수 생성

OOF 예측으로 다음 값을 계산한다.

- 예상 수익률
- 상승 확률
- 상대 강도
- 불확실성 점수
- 종합 신호 점수

### 11.3 시간순 70:30 분리

OOF 날짜를 시간순으로 나눈다.

```text
앞쪽 70% → 신호 가중치 튜닝
뒤쪽 30% → 백테스트 평가
```

무작위 분리가 아니므로 뒤쪽 평가 구간은 앞쪽 튜닝 구간보다 미래에 있다.

tune/eval 최소 고유 날짜 수가 부족하면 같은 OOF를 양쪽에 재사용하지 않는다. 이 경우 평가와 백테스트를 중단하고
`validation_split.status = "insufficient_data"`로 기록한다.

### 11.4 신호 가중치 튜닝

앞쪽 OOF에서 여러 가중치 조합을 비교한다.

```text
signal_score =
    return_weight × norm_return
  + up_prob_weight × up_probability
  + rel_strength_weight × rel_strength
  - uncertainty_penalty × uncertainty_score
```

상위 점수 종목의 실제 다음 날 평균 수익률이 가장 높은 조합을 선택한다.

### 11.5 백테스트

선택한 가중치를 적용한 뒤, 뒤쪽 OOF 구간에서 long-only top-k 백테스트를 실행한다.

## 12. OOF 예측과 최신 예측은 다르다

워크포워드 OOF 예측은 과거 성능 평가용이다.
사용자에게 제공하는 최신 다음 날 예측은 별도 최종 모델이 생성한다.

| 구분 | OOF 예측 | 최신 예측 |
|---|---|---|
| 예측 대상 | 과거 검증 구간 | 현재 가장 최신 날짜 |
| 학습 데이터 | 각 검증 날짜 이전의 과거 데이터 | 최근 학습 가능한 데이터 |
| 목적 | 검증, 튜닝, 백테스트 | 다음 날 예상 수익률 생성 |

현재 최종 모델은 기본적으로 최근 약 3년의 거래일을 사용한다.

```python
final_model_lookback_days = 252 * 3
```

```text
워크포워드 OOF 예측 ≠ 최신 운영 예측
```

## 13. 데이터가 부족한 경우

기본 설정으로 fold를 만들 수 없으면 파이프라인은 데이터 길이에 맞춰 설정을 줄여 다시 시도한다.

대략 다음 기준으로 기존 설정을 축소한다.

```text
최초 학습 크기 = 전체 고유 날짜의 약 60% 기준, 최소 60일
검증 크기      = 전체 고유 날짜의 약 20% 기준, 최소 20일
이동 간격      = 검증 크기의 약 절반 기준, 최소 20일
```

각 값은 원래 설정값보다 커지지 않는다.

그래도 OOF 예측을 만들 수 없으면 파이프라인은 오류를 발생시킨다.

```text
OOF predictions are empty. Increase data length or adjust training window.
```

## 14. 현재 구현에서 주의할 점

### 14.1 기본 검증 구간이 서로 겹칠 수 있다

기본값:

```text
test_size = 252
step_size = 126
```

검증 구간 크기보다 이동 간격이 작으므로 연속 fold의 검증 기간이 겹친다.

```text
Fold 1 검증: [────────────────]
Fold 2 검증:         [────────────────]
                    겹치는 기간 존재
```

따라서 원본 fold OOF에는 같은 `Date + Symbol`이 여러 번 포함될 수 있다.
현재 구현은 모든 fold 실행 후 같은 `Date + Symbol`의 예측값을 평균하여 고유 평가 행 하나로 집계한다.
실제 목표값이나 안정 컨텍스트 값이 서로 다르면 데이터 무결성 오류로 중단한다.

집계 OOF에는 다음 감사 정보가 남는다.

```text
oof_prediction_count
fold_ids
train_start_values / train_end_values
valid_start_values / valid_end_values
```

보고서의 `oof_policy.diagnostics`에는 원본 행 수, 고유 행 수, 중복 행 수, 중복 비율이 기록된다.

비중복 검증 구간이 필요하면 다음과 같은 정책을 고려할 수 있다.

```text
step_size >= test_size
```

또는 OOF 결합 후 중복 예측 처리 정책이 필요하다.

### 14.2 확률 보정과 평가 분리

현재 파이프라인은 고유 OOF를 먼저 시간순으로 분리한 뒤 앞쪽 tune OOF만으로 보정기를 학습한다.
뒤쪽 eval OOF는 학습된 보정기의 변환만 적용받고 최종 평가와 백테스트에 사용된다.

### 14.3 워크포워드 요약은 fold 단순 평균이다

각 fold의 표본 수가 달라도 현재 보고서에서는 fold별 성능 지표를 동일한 비중으로 평균한다.

### 14.4 Baseline 평가 기간이 다를 수 있다

워크포워드 모델 지표는 fold 검증 구간에서 계산한다.
현재 baseline 지표는 전체 특징 데이터에서 계산하므로 두 평가 기간이 정확히 같지 않을 수 있다.

## 15. 최종 정리

= 각 시험의 미래 검증 구간에서 모델이 만든 예측값 전체
