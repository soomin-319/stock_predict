## 0. 한눈에 보는 결론 (TL;DR)

1. **예측 엔진은 두 실행이 완전히 동일하다.** 같은 LightGBM 백엔드, 같은 111개 피처, **동일한 `feature_hash`(`1be64579…`)**, 같은 `random_state=42`. LLM provider는 예측 모델에 들어가지 않는다(가드레일: LLM 출력은 표시용).
2. 따라서 **두 실행의 예측 수익률(`predicted_return`) 차이는 provider 차이가 아니라 거의 전적으로 "종목 수(로컬 10 vs OpenAI 5)" 때문이다.** 종목 수가 학습표본·시그널 튜닝·확률 캘리브레이션·횡단면 순위를 바꾼다.
3. **provider의 진짜 차이는 4가지: 속도 · 비용 · 인프라 요구 · 데이터 거버넌스.**
   - 속도: OpenAI(클라우드)가 압도적으로 빠르다. 로컬 Gemma는 종목당 약 2.8분(이슈요약 단계).
   - 비용: Gemma는 API 과금 0(로컬 GPU·전력만), OpenAI는 종목별 호출 과금.
   - 인프라: Gemma는 16GB VRAM·`llama-server`·8001 포트·한글경로 버그 회피가 필요, OpenAI는 키만 있으면 된다.
   - 데이터: Gemma는 뉴스/공시 텍스트가 로컬에 머문다, OpenAI는 외부 API로 전송된다.
4. **이번 두 실행은 "깨끗한 A/B"가 아니다.** 종목 수가 다르고(10 vs 5), 뉴스 입력 가용성도 달랐다(네이버 429 + 캐시 상태). LLM 텍스트 품질의 직접 비교는 §6·§7 한계를 반드시 함께 읽어야 한다.

## 6. LLM이 실제로 다르게 기여한 부분 — 뉴스/이슈 요약 텍스트

예측에 안 들어가는(표시용) LLM 산출물에서만 provider 차이가 드러난다. 그런데 이번엔 **두 실행의 뉴스 입력 자체가 달라서** 직접 비교가 어렵다.

### 로컬 Gemma 실행 (공통 5종목)

- 공통 5종목의 표시용 이슈요약은 모두 **결정적 폴백 템플릿**이었다: 공시 = "유의미한 공시 신호가 약해…", 뉴스 = "수집된 뉴스가 없어…". 뉴스/공시 영향 점수 컬럼 = `-`.
- 즉 **공통 5종목에 대해서는 Gemma가 생성할 실제 뉴스 입력이 없었다.** (단, `news_impact_runtime.actual_mode=gemma`는 10종목 전체에서 수집된 raw_events 294건 중 유효행으로 Gemma 임팩트 경로가 동작했다는 의미다 — 다른 5종목에서 유효행이 있었던 것으로 보임.)

### OpenAI 실행 (공통 5종목)

- 카카오·NAVER·LG화학은 **캐시된 실제 뉴스 헤드라인 기반으로 OpenAI 이슈요약이 생성**됐다. 예: LG화학 뉴스요약에 "신용등급 결국 강등… 'AA'로 하향", "15조 R&D 승부수" 등 실제 기사 제목이 반영됨.
- 삼성전자·SK하이닉스는 뉴스가 없어 폴백 템플릿 처리.
- 뉴스/공시 **영향 점수**는 `rule_based` 폴백 결과(`+100.0점`)로 표시되고, 영향 참고 컬럼에 **`참고용·예측값 미반영`**으로 명시(가드레일 준수).

### 왜 입력이 달랐나 (교란요인)

- 두 실행 모두 신규 네이버 뉴스 수집은 **HTTP 429(Too Many Requests)**로 막혔다(로컬 raw_events 294건/OpenAI 144건은 캐시·기수집분 포함).
- 로컬 실행(12:12경)과 OpenAI 실행(13:07경)의 **캐시 상태가 달라서**, 공통 5종목 중 카카오·NAVER·LG화학의 뉴스가 OpenAI 실행 시점에는 캐시에 있었고 로컬 실행 시점 이슈요약에는 반영되지 않았다.
- 결론: **이건 "OpenAI가 뉴스를 더 잘 처리"한 게 아니라, 두 실행이 본 뉴스 입력이 달랐던 것**이다. provider 간 텍스트 품질 A/B로 보면 안 된다.

## 7. 공정한 비교의 한계 (교란요인 정리)

1. **종목 수 불일치 (10 vs 5)** — 가장 큰 교란. 학습표본·시그널 튜닝·확률 캘리브레이션·횡단면 순위를 모두 바꾼다. §4 예측 차이와 §5 지표 차이의 주원인. (로컬은 실제 10종목 유니버스 `data/universe_gemma_10.csv`로 실행됐고 `result/`·본 비교 수치는 모두 10종목 기준이다. 실행 *이후* `LOCAL_RUN.md` 기본 대상을 5종목으로 줄였으나, 동일 5종목 유니버스로의 깨끗한 5 vs 5 재실행은 **추후 진행 예정**이다.)
2. **뉴스 입력 가용성 불일치** — 네이버 429 + 캐시 상태 차이로, 뉴스임팩트가 한쪽은 `gemma`, 한쪽은 `rule_based`로 갈렸고 이슈요약 텍스트도 비대칭(§6).
3. **실행 시점 약 55분 차** — 캐시/외부 데이터 미세 차이.
4. **(코드·데이터 기준일은 동일)** — 같은 `feature_hash`, 같은 캘리브레이션 수정 적용, 같은 `input_as_of_date=2026-06-26`. 이 축들은 교란이 아니다.

## 8. 권고 — 언제 무엇을 쓸까 / 깨끗한 재비교 방법

**선택 가이드**

- **로컬 Gemma**: 반복·대량 실행, API 비용 0, 민감 데이터를 외부로 보내지 않아야 할 때, 오프라인 환경. 단 16GB VRAM·서버 기동·종목당 ~2.8분 감수.
- **OpenAI**: GPU가 없거나 빠른 턴어라운드가 필요할 때, 소수 종목. 단 종목별 과금·외부 전송·429 관리 필요.

**LLM 품질을 공정하게 비교하려면** (후속 제안)

1. **동일 종목 집합**으로 맞춘다(예: 둘 다 `data/universe_*_5.csv` 5종목).
2. **뉴스 캐시를 먼저 데운다** — 429를 피하도록 간격을 두고 뉴스 수집을 선행시켜, 두 실행이 **같은 뉴스 입력**을 보게 한다.
3. 그 상태에서 `--llm-config`만 gemma ↔ openai로 바꿔 연속 실행하면, **예측 지표(동일해야 정상) vs LLM 표시 텍스트(여기서만 차이)**를 분리해 평가할 수 있다.

> 핵심 재확인: 어떤 provider를 쓰든 **`predicted_return`은 바뀌지 않는다(가드레일).** provider 선택은 속도·비용·인프라·데이터 거버넌스·표시용 요약 품질의 문제이지, 매수/매도 신호 자체의 문제가 아니다.
