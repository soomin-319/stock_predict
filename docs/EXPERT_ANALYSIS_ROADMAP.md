# Stock Predict 전문가 진단 및 단계별 개선 로드맵

## 1) 현재 프로젝트 진단 (Executive Summary)

이 프로젝트는 다음 장점이 뚜렷합니다.

- **워크포워드(OOF) 기반 검증 체계**를 갖춰, 시계열 과최적화 리스크를 줄이려는 방향이 명확합니다.
- **회귀/분류/분위수 예측을 결합한 멀티헤드 구조**로 단일 포인트 예측보다 실전 활용성이 높습니다.
- **외부 시장 지표 + 진단 시각화 + 백테스트 아티팩트**가 연결되어 있어, 실험 결과를 해석하고 의사결정하기 좋은 형태입니다.

반면, 실전 배포/운영 관점에서 우선 개선할 필요가 있는 항목도 보입니다.

- **파이프라인 단일 파일 복잡도 증가**: `src/pipeline.py` 내부에 중복 정의가 다수 존재해 유지보수/리팩토링 난도가 높습니다.
- **데이터/모델 드리프트 감시 체계 부재**: 현재는 오프라인 리포트 중심이며, 운영 중 품질 저하를 조기 탐지하는 구조는 약합니다.
- **재현성/실험관리 고도화 필요**: 버전 고정, 데이터 스냅샷, 실험 추적(메타데이터 저장) 체계가 아직 약한 편입니다.

---

## 2) 아키텍처 관점 상세 분석

### A. 데이터 수집/특징 엔지니어링

- 기술지표와 외부지표를 함께 사용하는 구조는 합리적입니다.
- 외부지표 다운로드 실패 시 **graceful fallback** 처리(실패 시 빈 시리즈, 전체 실패 시 원본 반환)도 구현되어 안정성이 있습니다.

**리스크/개선 포인트**
- 외부 데이터 소스(yfinance) 장애/지연 시 성능과 일관성이 흔들릴 수 있으므로, 캐시 계층(예: 로컬 parquet + TTL)과 데이터 품질 체크가 필요합니다.
- 피처 중요도/기여도 추적이 자동화되어 있지 않아, 성능 저하 원인 분석 속도가 느려질 수 있습니다.

### B. 모델링/검증

- LightGBM 우선 + sklearn fallback 전략은 실행 환경 호환성 측면에서 장점입니다.
- 워크포워드 검증 및 OOF 생성은 시계열 문제에서 바람직한 기본기입니다.

**리스크/개선 포인트**
- 멀티헤드 구조가 유용하지만, 하이퍼파라미터 탐색/특징 선택/확률 보정이 정형화된 자동 루프로 완전히 묶여 있지는 않습니다.
- 현재 백테스트는 long-only top-k 규칙 기반으로 시작점은 좋으나, 체결 현실성(거래대금/유동성/슬리피지 동적화) 반영을 더 강화해야 합니다.

### C. 코드 품질/운영성

- 테스트 스위트가 존재하고 스모크 포함 검증 범위가 넓은 점은 강점입니다.

**리스크/개선 포인트**
- `src/pipeline.py` 중복 함수 정의는 기능 오류가 당장 없더라도, 향후 버그 유입 확률을 높입니다.
- 모듈 경계(데이터 준비/학습/평가/리포트/CLI)의 분리가 더 필요합니다.

## 3) 단계별 개선안 (실행 우선순위)

## Phase 0 (1~2주): 안정화/기초 체력 강화

목표: "지금 있는 기능을 안전하게 유지보수 가능한 상태"로 만듭니다.

1. **`src/pipeline.py` 리팩토링**
   - 중복 함수 제거 및 기능별 모듈 분리(`orchestration`, `io`, `reporting`, `postprocess` 등).
   - 동일 동작을 보장하는 회귀 테스트 추가.

2. **입출력 경로/설정 일관화**
   - 경로 강제 정책(`result/`)은 유지하되, 실행 환경별 override 정책을 명확히 문서화.

3. **기본 품질 게이트 도입**
   - `pytest -q` + lint(ruff/flake8) + format(black) + 간단 타입체크(mypy/pyright 일부) CI 구성.

**성과지표(KPI)**
- 리팩토링 후 테스트 전량 통과.
- 코드 중복(동일 함수 재정의) 0건.
- 신규 PR의 자동검증 100% 실행.

## Phase 1 (2~4주): 모델 성능/신뢰도 개선

목표: "성능뿐 아니라 예측 신뢰도를 측정 가능한 상태"로 고도화합니다.

1. **확률/불확실성 품질 평가 강화**
   - 현재 calibration plot 기반에서 Brier score, ECE(expected calibration error) 등 수치지표 추가.

2. **시계열 HPO 루프 도입**
   - Optuna 등으로 walk-forward 점수(예: 정보비율, 방향정확도, 손익지표) 최적화.

3. **피처 거버넌스 도입**
   - SHAP/Permutation 기반 중요도 리포트 자동 생성.
   - 저기여/고변동 피처 자동 감시.

**성과지표(KPI)**
- 기준 대비 OOF 방향정확도/Sharpe 개선.
- 확률 calibration error 감소.

## Phase 2 (3~6주): 실전형 백테스트/리스크 관리

목표: "백테스트 결과의 실전 이행 가능성"을 높입니다.

1. **거래비용 모델 현실화**
   - 고정 bps 외에 종목별 유동성/변동성 연동 슬리피지 도입.

2. **포트폴리오 제약 도입**
   - 섹터/종목 비중 제한, 최소 거래대금 필터, turnover cap.

3. **리스크 예산 기반 신호 집행**
   - 불확실성 기반 포지션 사이징(예: uncertainty score 연동).

**성과지표(KPI)**
- Max drawdown 개선.
- 거래비용 반영 후 성과 유지.
- 과도한 turnover 억제.

## Phase 3 (지속): 운영(MLOps) 체계화

목표: "모델을 계속 돌릴 수 있는 시스템"으로 전환합니다.

1. **실험/모델 레지스트리 도입**
   - 데이터 버전, 파라미터, 성능, 아티팩트(그래프/CSV) 추적.

2. **드리프트 모니터링**
   - 입력 분포/예측 분포/실현 성능 모니터링 및 알람.

3. **배포 전략 표준화**
   - shadow/canary 방식으로 모델 전환 리스크 최소화.

**성과지표(KPI)**
- 모델 롤백 시간 단축.
- 성능 저하 조기 탐지율 향상.

## 4) 30/60/90일 실천 계획 (현실적인 실행안)

### 0~30일
- 파이프라인 중복 코드 제거 + 모듈 분리 착수.
- CI에 lint/format/test 게이트 설치.
- 리포트 스키마 고정(JSON schema) 및 문서화.

### 31~60일
- HPO + 확률 calibration 수치평가 도입.
- 피처 중요도/데이터 품질 리포트 자동화.
- 백테스트 제약(유동성/turnover cap) 1차 적용.

### 61~90일
- 실험/모델 레지스트리, 드리프트 모니터링 연결.
- 배포(주간/일간) 스케줄 자동화 및 장애 대응 플레이북 정리.

---

## 5) 결론

현재 프로젝트는 "연구용 베이스라인"을 넘어서 "준운영 단계"로 진입할 수 있는 기반이 이미 있습니다.
가장 큰 병목은 **코드 구조 안정화(특히 파이프라인 중복 제거)**와 **운영 지표화(MLOps)** 입니다.

따라서 권장 전략은 아래 한 줄로 요약됩니다.

> **Phase 0(안정화) → Phase 1(성능/신뢰도) → Phase 2(실전 리스크) → Phase 3(운영 자동화)** 순으로, 기능 확장보다 먼저 구조 안정화를 완료하세요.
